## 2025-02-12 - Missing API Authentication for Paid External Services Orchestrator
**Vulnerability:** The internal API (`api/server.py`) endpoints (`/extract` and `/check`) were publicly accessible without any authentication. Since these endpoints orchestrate requests to paid and rate-limited external services (e.g., OpenRouter, Binance API, Perplexity, RPCs), unauthorized access could easily lead to API quota depletion, financial loss, and Denial of Service (DoS) for the application. Furthermore, error messages directly leaked raw internal exceptions, exposing sensitive system information.
**Learning:** Internal APIs that consume external/paid resources must always implement their own authentication layer, even if they are only intended to be called by trusted internal clients (like the Telegram Bot or Frontend). Never assume an API is safe just because its clients are internal; if it's exposed, it can be exploited. Similarly, unhandled exception traces pose an information leakage risk and should never be bubbled up to the client in plain text.
**Prevention:** Implement an `APIKeyHeader` check (e.g., `X-API-Key`) against a securely stored environment variable (`SAFE_SENTINEL_API_KEY`). Use `secrets.compare_digest` to prevent timing attacks. Enforce a fail-secure configuration where the API returns a 500 status code if the API key is not configured in the environment, rather than defaulting to open access. Implement robust exception handling that logs full stack traces internally using `logger.error(..., exc_info=True)` but returns a generic error message (e.g., "Internal Server Error") to the client.
